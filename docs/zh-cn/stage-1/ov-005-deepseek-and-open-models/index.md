## 本节导入

DeepSeek 受到关注，不应只归结为“又出现了一个聊天模型”。它让更多团队同时看到三件事：推理模型可以通过训练与测试时计算改善复杂任务表现；开放权重让模型能够进入更多部署环境；模型选择也从“谁的榜单分数最高”变成质量、成本、隐私与工程能力的综合决策。

这里的结论必须绑定具体版本。DeepSeek-V3、DeepSeek-R1、蒸馏模型和后续版本的能力、许可证、上下文长度与部署要求并不相同；任何价格、速度或榜单数字也只在对应日期、硬件和评测设置下成立。

## 核心内容

### 推理模型改变了什么

普通生成模型倾向于直接预测下一个 token。推理模型则通过训练，使模型在数学、代码、规划等任务上更善于分解问题、检查中间结果并在推理阶段投入更多计算。它并不意味着模型拥有稳定的逻辑真理，也不保证每一步解释都可信；复杂任务仍要用可执行结果、外部工具或独立证据验证。

DeepSeek-R1 论文描述的路线包含冷启动监督数据、面向推理的强化学习、混合数据监督微调和后续强化学习等阶段。强化学习用于奖励可验证的正确结果和合适行为，监督微调用高质量示例稳定语言与任务表现，蒸馏则把较大模型产生的能力迁移到较小模型。学习这些环节的价值，在于理解“推理模型”是训练系统的产物，不是把提示词改成“请一步一步想”这么简单。

### “开源模型”要拆成四个问题

日常讨论常把可下载模型统称为开源，但工程决策要继续追问：模型权重是否可得？推理与训练代码是否公开？训练数据和处理方法是否可追溯？许可证是否允许修改、商用、再分发和生成衍生模型？

只有权重可下载，通常更准确地称为**开放权重模型**。它仍然很有价值：团队可以在自己的基础设施上运行、量化、评测和微调；但它并不自动带来训练透明度，也不代表所有用途都无条件允许。部署前应保存具体版本、权重校验值、许可证文本和获取日期。

### API、托管与本地部署的取舍

调用外部 API 上手快、无需管理 GPU，适合快速试验和波动负载；托管专属实例能增加网络与容量控制；本地部署可以让数据、日志和模型运行环境留在组织边界内，却要自己承担硬件、推理框架、容量规划、漏洞修复和升级评测。

判断方案时至少测六项：目标任务质量、端到端时延、峰值吞吐、单位任务总成本、敏感数据暴露面和持续运维能力。参数量不是硬件需求的唯一决定因素；权重量化、上下文长度、并发数和 KV cache 都会占用显存。所谓“更便宜”也不能只比较 token 单价，还要加上检索、审核、重试、人工接管和基础设施成本。

### 模型评测必须贴近真实任务

公开榜单适合形成候选列表，最终选择应使用自己的代表性任务集。知识问答要检查引用是否支持结论；代码任务要运行测试；结构化抽取要核对字段；推理任务要验证最终答案和关键约束；高风险任务还要测拒答、越权和敏感信息泄露。

评测结果应记录模型版本、推理参数、提示模板、工具、数据集版本和运行日期。模型升级后重新跑同一套基线，才能知道变化来自模型本身还是调用配置。

## 案例与图解

### 内部制度问答：不是“装上本地模型”就完成

一家企业希望员工查询内部制度，初看适合本地部署。团队先把 100 个真实问题整理成评测集，要求回答同时给出制度名称、章节和生效日期。方案采用权限过滤后的文档检索，再把证据交给模型生成答案；找不到可靠依据时明确转人工。

试点同时比较外部 API、专属托管和本地开放权重模型。最终选择不由单次演示决定，而由引用正确率、无依据回答率、响应时延、单位任务成本、运维工时和敏感数据边界共同决定。即使模型在本地，文档权限、日志脱敏和过期制度下线仍然必须单独实现。

## 动手实践

1. 选择一个真实任务，写出 20 条最小评测集，并定义“答对”的证据，而不是只给主观评分。
2. 为同一任务比较外部 API、托管实例和本地部署，列出质量、时延、成本、数据、运维六栏假设。
3. 找到一个拟用模型的许可证，分别回答：能否商用、能否改模型、能否再分发、是否有额外使用限制。
