## 学习目标

- 区分 G1 人形机器人与 Go2 四足机器人在自由度、平衡、跌倒风险和控制接口上的差异。
- 用节点、话题、服务、参数和 launch 文件画出 ROS2 控制链路。
- 用无需真机的固定消息与 trace 演练，验证“环境确认—只读检查—受限动作—超时停止—记录”的实验闭环。
- 根据电源、网络、节点、话题、坐标和硬件状态逐层定位异常。

## 本节导入

ROS2 让传感器、控制器、可视化和应用程序通过明确接口协作，但“节点都启动了”并不代表机器人适合动作。真实设备还受到电量、网络时延、地面摩擦、关节温度、人员距离和急停可用性的约束。

本节主课件以 Go2 基础功能包为背景，覆盖键盘控制、`Twist` 消息桥接、机器人模型可视化和驱动包。公开讲义不要求学习者连接真机，而是先用固定消息契约和离线 trace 学会设计、检查与停止。G1 可以沿用相同的检查方法，但不能照搬 Go2 的动作接口：人形机器人自由度更高、平衡与跌倒后果更复杂，真机操作必须使用对应型号与固件的官方接口，并在具备保护条件的现场进行。

## 核心内容

### 实训前置：先确认“谁能让它停下”

上电前清空活动区域，检查地面、关节外观、电池锁定、遥控器或急停、无线与有线网络。指定一名操作员负责命令，一名安全观察员负责急停；两人都要知道失联、异常姿态、异响、过热和人员进入安全区时如何停止。

软件环境记录操作系统、ROS2 发行版、工作区版本、机器人型号、固件和网络配置。不同版本的包名、话题名和参数可能变化；任何真机课程都应在开始前发布可访问的版本清单、接口表和急停说明，并以对应版本的厂商文档为准，不把课堂截图当成永不过期的接口规范。

### ROS2 图谱：用固定离线演练理解接口

ROS2 **节点**承担功能，**话题**传递连续消息，**服务**处理一次请求与响应，**参数**配置节点行为，**launch** 文件把多个节点按依赖启动。为了让没有设备和 ROS2 环境的学习者也能完成练习，本节固定使用下面这张纸面图谱：

| 节点 | 读取 | 输出 | 通过条件 |
| --- | --- | --- | --- |
| `teleop_demo` | 学习者的前进、转向、停止选择 | `/cmd_vel_demo`，类型为 `Twist` | 只使用规定轴，数值不超过限幅 |
| `state_replay_demo` | 固定状态序列 | `/robot_state_demo`，状态为 `ready`、`stopped` 或 `fault` | 状态带时间且按题目顺序变化 |
| `safety_bridge_demo` | 速度命令与机器人状态 | `/robot_command_demo` | 超时、非 `ready` 或越界时只输出零命令 |
| `trace_recorder_demo` | 上述三个话题 | 时间、输入、判断、输出和原因 | 能解释每次放行或拦截 |

这不是某一型号机器人的真实接口表，而是一份用于学习安全契约的固定演练。进入真机前，再把演练名称逐项映射为对应 SDK 和固件的真实接口，并重新核对消息类型、单位、频率、权限和停止语义。

### 键盘控制：先定义可验证的消息契约

键盘控制节点读取按键，把前进、后退、转向和停止映射为机器人命令。可靠实现应限制最大线速度和角速度，按键松开或输入超时后自动发送零速度，并把退出键与急停行为写在界面上。

本节离线演练采用固定契约：前进命令为 `linear.x = 0.05`，左右转向只使用 `angular.z` 且绝对值不超过 `0.10`，其余分量始终为零；连续 `0.50` 秒没有新命令时，桥接必须输出全零；机器人状态不是 `ready` 时，任何非零输入都被拒绝并记录原因。学习者可以直接依据这些数字判断 trace，无需安装软件或连接机器人。

真正实现时，消息定义、构建方式和可执行入口应来自公开且固定版本的工作区说明；在这些信息缺失时，不应凭课件截图猜包名或接入真机。

### Twist 桥接：在通用接口与设备接口之间转换

导航与键盘工具常发布 `geometry_msgs/msg/Twist`，其中线速度和角速度使用通用字段；设备驱动可能使用厂商消息或专用 API。桥接节点订阅 `Twist`，做限幅、坐标约定和超时检查，再转换为设备命令。

桥接不是简单“字段复制”。必须确认前进轴、旋转方向、单位、更新频率、停止语义和失联策略。测试顺序是零命令、极低速度短脉冲、停止、单轴移动，再进入组合动作；每一步都观察状态回读和安全区。

### 模型可视化：URDF、TF 与实机要一致

URDF 描述连杆、关节、惯性和视觉模型，`robot_state_publisher` 根据关节状态发布 TF，RViz 用来检查模型姿态与坐标关系。模型可视化通过不等于真机状态正确，但它能提前暴露关节名、父子关系、坐标轴和资源路径错误。

检查固定姿态时各关节是否方向一致，移动后模型是否跟随，`base`、里程计与世界坐标是否形成连通 TF 树。不要为了“画面看起来对”而在 RViz 中临时补错坐标，应回到 URDF、标定或驱动配置修正。

### G1 与 Go2：同一协议，不同风险包络

Go2 四足结构支撑面较稳定，适合教学中从速度命令、状态读取和导航接口入门；G1 人形平台的关节数、质心高度和接触切换更复杂，跌倒可能伤及设备和周围人员。两者可共享 ROS2 的观测、记录和诊断方法，但动作命令、限幅、保护架、起身策略和现场批准必须分别设计。

### 诊断顺序：不要一上来改代码

出现异常时先停止动作并保持现场安全，然后按“电源与急停—物理网络—设备可达—驱动进程—节点—话题类型与频率—TF 与参数—应用逻辑—硬件故障”逐层排查。每层都留下可观察证据，避免同时改网络、参数和代码后无法判断哪项真正起作用。

## 案例与图解

### 无需真机的固定 trace 验收

按上面的契约判断以下记录，预期结果是唯一的：

| 时间（秒） | 新输入 | `robot_state` | 桥接应输出 | 原因 |
| --- | --- | --- | --- | --- |
| 0.00 | 无 | `ready` | 全零 | 尚无速度命令 |
| 0.10 | `linear.x = 0.05` | `ready` | `linear.x = 0.05` | 状态正常且未越界 |
| 0.30 | `linear.x = 0.05` | `ready` | `linear.x = 0.05` | 新命令刷新超时计时 |
| 0.81 | 无 | `ready` | 全零 | 距上次命令 0.51 秒，已超时 |
| 1.00 | 无 | `fault` | 全零 | 故障状态禁止动作 |
| 1.10 | `linear.x = 0.05` | `fault` | 全零并记录拒绝 | 非 `ready` 状态不得放行 |

如果学习者在 0.81 秒或 1.10 秒仍输出非零命令，说明安全桥接不合格。这个演练验证的是控制契约和停止逻辑，不表示任何真实机器人已通过安全测试。

## 动手实践

1. 根据固定图谱，画出四个节点与三个话题的方向，并标出哪一处拥有最终放行权。
2. 独立计算固定 trace 的六行预期输出，再与表格答案比较；任何一行不一致都要写出错误原因。
3. 新增两个测试：一个越过线速度限幅，一个在 `stopped` 状态发出转向命令，写出桥接应输出什么。
4. 设计一份未来真机实验记录模板：设备与软件版本、人员分工、接口映射、启动顺序、动作、回读、异常、停止和结论。没有完成接口映射与现场批准时，模板保持“禁止真机动作”。

## 课后复盘

- 为什么控制前必须先检查只读状态与消息频率？
- `Twist` 是通用接口，为什么仍不能直接假设所有机器人行为一致？
- RViz 模型正确与真实机器人安全之间还隔着哪些环节？
- G1 和 Go2 可以复用哪些 ROS2 方法，哪些控制与安全设置不能复用？
- 遇到异常时，证据化诊断为什么比同时改多个配置更可靠？
