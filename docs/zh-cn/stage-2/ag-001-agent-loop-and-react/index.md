## 学习目标

- 画出 Observe—Reason/Plan—Act—Observe—Evaluate 循环，并说明每一步的输入和输出。
- 用 ReAct 的“推理—行动—观察”交错方式解释智能体如何根据环境反馈修正下一步。
- 为一个任务写出成功、受阻、超限、风险升级和人工终止等停止条件。
- 阅读一段最小 trace，判断智能体在哪一步偏离目标以及应从哪里恢复。
## 知识导图

智能体的关键不是多说一步推理，而是能在环境反馈中持续选择可验证的行动。

- **三类系统：** 先区分语言模型、固定工作流与能够自主选行动的智能体。

- **ReAct 循环：** 让观察、推理、行动和新反馈交错发生，逐步缩小不确定性。

- **停止与复盘：** 用完成、阻塞、风险和预算条件决定何时停止，并保留执行轨迹。

## 本节导入

一次大模型调用通常是“给出输入，得到输出”。智能体则要在变化的环境中持续工作：它读取当前状态，选择一个动作，取得新证据，再决定继续、改道还是停止。因此，智能体的核心不只是“更会回答”，而是一个能够反复观察和校正的执行循环。

这个循环也解释了智能体为什么会成功或失败。观察遗漏关键约束，计划就会建立在错误前提上；动作没有权限边界，错误就可能进入真实系统；执行后不评估，局部偏差会沿任务链不断放大。学习 Agent Loop，就是学习如何让每一步都可验证、可停止、可恢复。

## 核心内容

### 先区分模型、工作流与智能体

| 形态 | 下一步由谁决定 | 是否读取环境反馈 | 适合任务 |
| --- | --- | --- | --- |
| 单次模型调用 | 调用者预先决定 | 通常不读取 | 摘要、改写、分类等边界清晰的任务 |
| 确定性工作流 | 程序按固定顺序决定 | 按预设分支读取 | 审批流、固定报表、标准数据处理 |
| 智能体 | 模型与规则共同决定 | 持续读取并调整 | 步骤无法完全预知、需要工具探索的任务 |

并非所有问题都需要智能体。固定流程能稳定解决的任务，使用工作流通常更便宜、更容易测试。只有当任务需要根据中间结果选择路径、调用不同工具或处理开放环境时，Agent Loop 才真正产生价值。

### Observe：把事实与假设分开

观察不是把所有上下文一次性塞给模型，而是获取当前决策所需的证据。一次可靠观察至少回答五个问题：目标是什么、已经完成什么、仍缺什么、可以使用哪些工具、当前有哪些约束。工具返回值、文件内容、测试结果和用户确认属于事实；“大概已经安装”“应该有权限”只是待验证假设。

好的观察会保留来源和时间。例如，“测试通过”应关联测试名称、运行时间和结果；“文件存在”应记录路径和读取状态。环境可能变化，旧观察不能自动当作当前事实。

### Reason / Plan：选择最小可验证的下一步

Reason 关注“当前证据说明什么”，Plan 关注“接下来做什么”。简单任务可以只选择一个下一动作；长任务则需要把目标拆成阶段、依赖和验收点。计划不是不可修改的剧本，而是基于当前状态形成的暂定路线。

一个好的下一步应当足够小：动作完成后，系统能明确判断状态是否改善。例如，与其计划“修好网站”，不如先“复现故障并保存错误信息”，再决定检查数据、界面还是网络。小步行动缩短了错误传播距离，也使恢复更容易。

### Act：以最小权限改变状态

行动通过工具发生，例如查询资料、读取文件、调用接口、运行测试或修改记录。每个动作都应带有明确参数、预期结果、权限级别和失败处理。只读动作与写入动作要分开；可逆修改与外发、删除、授权等高影响动作也要分开。风险越高，越应在行动前设置人工确认。

### Observe again：把工具结果变成新状态

工具调用完成不等于任务完成。智能体要读取真实返回：是否成功、返回了什么、是否只完成一部分、是否出现警告、环境是否被改变。超时不能被解释为成功，空结果也不能被自动补写成“没有问题”。这一次观察会成为下一轮推理的依据。

### Evaluate：对照标准决定继续还是停止

Evaluate 把当前结果与验收标准比较。它既检查最终目标，也检查过程约束：结果是否完整，证据是否足够，是否使用了允许的数据和权限，是否引入新的风险。如果不满足标准，系统应指出差距并选择修正动作；如果已经满足，就应停止，而不是为了“更好”无限循环。

### ReAct：让推理与行动交错发生

ReAct 的关键思想，是让任务求解在“形成行动依据—采取行动—读取观察”之间交错进行。模型不必在一开始猜出全部步骤，而是用环境反馈更新判断。例如，检索未找到目标资料时，它可以调整关键词；测试失败时，它可以依据错误定位下一处检查点。

工程系统不需要保存或暴露模型的全部内部思维。trace 更适合记录简洁的决策摘要、所依据的证据、工具参数和结果，使过程能够审计，同时避免把敏感上下文或冗长推理写入日志。

### 停止条件：完成只是其中一种

| 停止类型 | 触发示例 | 系统应输出什么 |
| --- | --- | --- |
| 成功停止 | 验收项全部通过，证据齐全 | 结果、证据、影响范围 |
| 受阻停止 | 缺少文件、权限、密钥或必要选择 | 已完成部分与最小缺口 |
| 超限停止 | 达到时间、费用、工具调用或循环次数上限 | 当前进度与继续成本 |
| 风险升级 | 即将外发、删除、授权，或结果低置信度 | 待审核动作与风险说明 |
| 异常停止 | 同类失败重复出现、状态没有变化 | 失败序列与恢复建议 |
| 人工终止 | 用户取消或改变目标 | 已产生的状态变化与清理结果 |

如果没有停止条件，智能体可能反复调用同一工具、在错误方向上持续消耗资源，甚至把不确定结果当作完成。停止规则应在执行前定义，并在每轮 Evaluate 中检查。

### Trace：把循环变成可复盘证据

一条最小 trace 可以包含：`task_id`、`step_id`、时间、当前目标、观察来源、决策摘要、动作名称、输入参数摘要、权限级别、工具结果、评测结论、状态变化、下一步和停止原因。敏感字段应脱敏，原始凭据不应进入 trace。

trace 的价值不是“记录得越多越好”，而是回答三个问题：系统为什么采取这一步、这一步实际发生了什么、失败后从哪里恢复。只有能支持这三个问题的记录，才真正服务于调试和责任追踪。

### 从“整理会议纪要”看一次完整循环

假设目标是从三份会议纪要中生成待办清单，并要求每条待办都有负责人和截止时间。

1. **Observe：**确认三份文件均可读取，记录输出格式，并发现其中一份纪要没有明确截止时间。
2. **Reason / Plan：**先抽取可确认的任务，再把缺失字段列为待澄清项，不凭空补日期。
3. **Act：**调用文档读取工具，逐份提取“事项—负责人—截止时间—原文位置”。
4. **Observe again：**工具成功读取三份文件，共得到八条候选事项，其中两条负责人表述冲突。
5. **Evaluate：**六条满足标准；两条缺少可靠字段，不能进入最终待办。
6. **Stop：**以“部分完成、等待确认”停止，交付六条待办、两条问题和对应原文证据。

这个例子中，停止并不意味着失败。系统拒绝猜测，并准确指出缺口，反而比输出一份看似完整但含有虚构信息的清单更可靠。

## 动手实践

选择一个“查询两天内的天气并给出出行建议”或“检查一份表格中的缺失字段”任务，完成以下练习：

1. 写出初始观察清单，至少区分三项事实和两项待验证假设。
2. 把任务画成 Observe—Reason/Plan—Act—Observe—Evaluate 循环，并为每个动作写出预期证据。
3. 设计至少四类停止条件，其中必须包括成功停止、受阻停止和风险升级。
4. 用表格模拟五条 trace 记录，故意加入一次工具失败，再标出系统应如何恢复。

验收时，不只检查最终答案，还要检查每个结论能否回到观察证据，失败是否被如实记录，达到停止条件后是否立即结束。

## 课后复盘

- 为什么“工具返回成功”不能直接等同于“任务完成”？
- ReAct 与“一开始生成完整计划再执行”相比，适合处理什么变化？
- 哪些任务使用确定性工作流比使用智能体更合适？
- 为什么 trace 应记录决策摘要，却不应无边界保存敏感上下文和全部内部推理？
- 当任务只能完成一部分时，怎样的停止输出仍然是高质量交付？
