## 学习目标

- 把一个模糊想法改写成包含用户、行为、约束和验收标准的开发任务。
- 完成需求—基线—生成—运行—观察—修复—测试—审阅—提交的最小项目闭环。
- 根据错误证据定位问题，避免在没有复现和验证时反复重写代码。
- 识别 AI 生成代码中的安全、隐私、依赖、许可证和可维护性风险。

## 本节导入

AI Coding 是利用模型理解代码、生成修改、解释错误和辅助测试。Vibe Coding 更强调用自然语言快速表达意图，在运行结果与直觉反馈之间连续迭代。它们都能显著降低从想法到原型的门槛，但“代码已经生成”不等于“产品已经完成”。

真正可靠的开发仍要面对同样的工程事实：需求可能含糊，运行环境可能不同，依赖可能失效，交互可能在边界条件下出错，生成代码也可能引入安全问题。AI 改变的是实现速度和交互方式，不会取消测试、版本控制和责任审阅。本节用一个不依赖具体产品的最小项目走完闭环；学习者可以使用任意图形化编辑器、命令行助手或网页式代码工具替换示例中的工作界面。

## 核心内容

### 第一步：把“想做什么”变成可验收需求

“做一个好看的待办应用”没有说明谁使用、要完成什么、数据如何保存、什么叫成功。可执行需求可以写成：

> 为个人学习者制作一个浏览器端待办清单。用户可以新增非空任务、标记完成、删除任务；刷新页面后数据仍保留；空输入不会创建任务；键盘可完成主要操作。项目不需要账号、服务器和外部数据库。

然后把它变成验收清单：首次打开显示空状态；输入文字并提交后出现一条任务；完成状态可切换；删除后任务消失；刷新后状态保持；空白输入被拒绝；窄屏下按钮不遮挡文字。需求越能通过操作观察，模型越容易交付正确结果。

### 第二步：确认环境与基线

开始修改前，应确认项目目录、启动方式、当前页面和已有错误。如果是已有项目，先运行一次并记录基线：哪些功能已经工作，哪些测试通过，工作区是否有未保存修改。这样才能区分“原来就存在的问题”和“本轮修改引入的问题”。

最小静态项目可以只有 `index.html`、`style.css` 和 `app.js` 三个文件，通过本地静态服务器打开。真实项目则应优先遵循项目已有的安装、运行和测试方式，不要为了方便自行替换技术栈。

### 第三步：给 AI 一张清楚的任务卡

一次有效任务卡通常包含：目标、现状、允许修改的范围、必须保留的行为、技术约束、验收步骤和期望输出。例如：

```text
目标：完成待办清单的新增、完成、删除和本地保存。
范围：只修改 index.html、style.css、app.js。
约束：不引入第三方依赖；数据仅保存在浏览器本地；保持键盘可操作。
验收：按七条验收清单逐项验证，并说明改了什么、测试了什么。
执行方式：先阅读现有文件，再提出最小修改；每轮只处理一组相关问题。
```

任务卡不是越长越好。关键是让目标与边界可判断，使 AI 知道何时应停止、何时必须询问。

### 第四步：小步生成，立即运行

先实现一条端到端的“最窄可用路径”：页面能打开，新增任务能显示。运行并确认后，再加入完成、删除和保存。每次修改只包含一组相关行为，并在版本控制中留下可回退点。

小步的价值在于缩小故障范围。如果一次同时改结构、样式、数据模型和存储，出现错误时很难知道是哪一处造成的。让 AI 一次生成大量代码看似更快，但调试和审阅成本可能抵消节省的时间。

### 第五步：用证据调试，而不是凭感觉重写

遇到“点击没有反应”，先稳定复现，再收集证据：操作步骤是什么，控制台是否报错，事件是否触发，数据是否变化，页面元素是否存在。把原始错误、相关代码和预期行为交给 AI，请它解释最可能的原因并提出最小修复。

修复后必须重复原复现步骤，同时检查相邻功能。一个错误消失，可能只是被隐藏；例如删除按钮恢复后，本地保存却没有同步。调试的完成条件是原因被验证、修复通过、没有明显回归，而不是页面暂时不报错。

### 第六步：分层测试

最小项目也应有三层检查：

1. **静态检查：**文件能解析，没有明显语法错误和无效引用。
2. **行为检查：**按验收清单操作正常，包括空输入、重复点击、刷新和窄屏等边界。
3. **回归检查：**修改新功能后，已有功能仍然可用。

有自动化测试时，应先运行与修改最相关的测试，再运行完整测试。没有自动化测试时，也要把手工步骤和结果记录下来，使他人能够复现。

### 第七步：审阅 AI 生成代码

审阅至少覆盖：是否只修改了任务范围内的文件；是否把密钥或个人数据写入代码；是否新增未知依赖；输入是否被安全处理；错误是否被静默吞掉；代码命名和结构是否便于维护；引用的素材和代码是否允许使用。

对外发布前还要确认许可证、隐私说明和部署配置。即使页面“看起来能用”，未经审阅的生成代码也不应直接进入生产环境。

### 第八步：提交可复现结果

一次完整交付应包含：实现结果、变更摘要、验证命令或操作、已知限制和版本记录。提交信息描述一个完整意图，例如“实现待办新增与本地保存”，不要把多个无关修改混在一起。若后续发现问题，可以准确回退或比较。

## 案例与图解

### 待办清单的最小闭环

| 阶段 | 具体动作 | 通过证据 |
| --- | --- | --- |
| 需求 | 固定七条可操作验收项 | 每条都能观察成功或失败 |
| 基线 | 启动项目并记录现状 | 页面截图、当前错误、版本状态 |
| 生成 | 先完成新增任务 | 新任务出现在列表中 |
| 迭代 | 加入完成、删除与保存 | 每组行为分别验证 |
| 调试 | 用复现步骤和错误定位 | 原因明确，最小修复有效 |
| 测试 | 正常、边界、刷新、窄屏 | 验收清单全部有结果 |
| 审阅 | 检查范围、安全与依赖 | 没有密钥、越界修改和未知依赖 |
| 提交 | 保存可回退版本 | 他人可按说明重新运行 |

如果某一步失败，就回到最近一个已验证状态，而不是让 AI 重新生成整个项目。Vibe Coding 的速度来自快速反馈，不来自跳过证据。

## 动手实践

完成上述待办清单，或选择功能规模相同的“学习计时器”。全过程保留一份简短开发记录：

1. 写出用户故事、范围外事项和至少七条验收标准。
2. 启动基线并记录现状，创建第一个可回退版本。
3. 分三轮实现核心路径、边界处理和视觉适配，每轮都运行验证。
4. 人为制造一次错误，记录复现步骤、原始错误、原因、最小修复和回归结果。
5. 审阅所有改动，确认没有秘密信息、未经说明的依赖和范围外修改。
6. 让另一位学习者只依据你的运行说明打开项目并完成验收。

最终交付不是聊天记录，而是可运行项目、验收结果、已知限制和可追溯版本。

## 课后复盘

- 为什么自然语言描述越流畅，越不代表需求已经可验收？
- 小步修改如何降低 AI 生成代码的调试成本？
- “错误不再出现”和“问题已经修复”之间还差哪些证据？
- 哪些审阅项不能仅交给生成代码的同一个模型完成？
- 在什么情况下，快速原型应停止扩展并转入重新设计？
