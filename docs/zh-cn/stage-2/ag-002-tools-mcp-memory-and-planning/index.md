## 学习目标

- 为一个工具写出名称、用途、输入、输出、权限、失败语义和幂等性要求。
- 说明 MCP 中 host、client、server 以及 tools、resources、prompts 的职责边界。
- 区分当前上下文、任务短期状态和长期记忆，并为记忆设置来源、范围和失效条件。
- 把长任务拆成可验收步骤，并正确处理无权限、工具异常和部分完成。

## 本节导入

语言模型能生成文字，但不能仅凭文字读取最新库存、查询内部资料或修改日程。智能体要进入真实业务，必须通过工具与外部系统交互。工具把“模型建议做什么”转换为“系统实际执行了什么”，也把权限、数据质量和错误恢复带入了智能体设计。

与此同时，长任务需要管理信息。当前对话里出现的内容不等于任务状态，任务状态也不等于长期记忆。如果把三者混在一起，系统会遗忘关键进度、引用过期偏好，或把一次偶然结果永久保存。本节把工具契约、MCP、状态、记忆和规划放在同一条执行链上理解。

## 核心内容

### 工具契约：先约定，再调用

工具不是一段模糊的自然语言说明，而是一份可检查的契约。至少应包含以下部分：

| 字段 | 要回答的问题 | 示例 |
| --- | --- | --- |
| 名称与用途 | 工具做什么、不做什么 | `search_orders` 只查询订单，不修改状态 |
| 输入模式 | 哪些参数必填、类型和范围是什么 | `order_id` 为非空字符串，`limit` 为 1—50 |
| 输出模式 | 成功时返回哪些稳定字段 | 状态、更新时间、数据来源 |
| 失败语义 | 调用者如何区分错误 | 未授权、未找到、限流、超时、参数无效 |
| 权限与影响 | 谁可调用，会改变什么 | 只读；或写入后需要审核 |
| 幂等性 | 重试会不会产生重复副作用 | 同一请求标识重复提交只生效一次 |

描述越含糊，模型越容易选错工具或补造参数。对于写入型工具，还要提供“预览”或“草稿”能力，让智能体先展示将要发生的变化，再进入真正执行。删除、外发、付款、授权等高影响动作不应与普通查询共享同一权限级别。

### 工具结果：失败必须是可读状态

工具调用可能出现四类结果：成功、部分成功、可重试失败和不可重试失败。超时或网络中断通常只能说明“结果未知”，不能直接再次写入，否则可能重复创建记录。参数错误应先修正输入；未授权应停止并请求合适权限；业务规则拒绝则需要改变方案，而不是重复调用。

智能体要读取结构化错误码、错误信息和是否可重试，而不是从一句模糊文本中猜测。系统还应保存调用标识，便于确认一次写入究竟是否已经发生。

### MCP：连接能力的通用边界

Model Context Protocol（MCP）提供了一套连接 AI 应用与外部能力的协议。可以把它理解为三层：

- **Host：**承载用户体验和安全策略的 AI 应用，决定哪些连接可用、哪些信息可以进入模型上下文。
- **Client：**host 内与某个 server 建立会话的协议端，负责能力发现、消息交换和连接生命周期。
- **Server：**暴露特定数据或操作能力的一端，可以是本地进程，也可以是远程服务。

MCP server 可以提供三类常见能力：

- **Tools：**可被模型选择调用的操作，例如查询数据库或运行计算；调用前仍需遵守 host 的授权策略。
- **Resources：**可供应用读取并加入上下文的数据，例如文档、配置或数据库记录。
- **Prompts：**server 提供的可复用提示模板，通常由用户或应用显式选择，用来组织某类任务的输入。

协议解决的是“如何描述、发现和调用能力”，并不会自动解决业务授权、数据合规或结果正确性。同一个工具即使符合协议，也可能因为用户身份、资源范围或组织政策而被拒绝。

### 上下文、短期状态与长期记忆

这三类信息的生命周期不同：

1. **当前上下文**服务于眼前一次决策，包括用户本轮要求、刚取得的工具结果和相关资料片段。它容量有限，任务推进时会变化。
2. **短期状态**记录一个任务的可恢复进度，例如目标、已完成步骤、中间产物、待处理错误和下一步。即使会话中断，任务也应能从状态恢复。
3. **长期记忆**保存跨任务仍有价值的信息，例如用户明确确认的偏好、稳定业务规则和反复验证的经验。它必须有来源、适用范围、更新时间和失效方式。

长期记忆不能替代事实查询。“用户上次偏好上午开会”不代表这次一定如此；“某接口去年这样使用”也不代表当前仍有效。容易变化且查询成本低的事实，应在当前任务中重新验证。

### 规划：维护可更新的任务图

规划的作用是把目标转成可执行且可验收的步骤。每一步应写清输入、动作、产出、依赖和通过条件。存在依赖时，计划更像任务图而不是线性清单：数据未取得，分析步骤就不能开始；审批未通过，外发步骤就必须保持阻塞。

计划要随着观察更新，但变更必须有理由。如果工具返回的真实结构与预期不同，应修订后续步骤；如果只是执行困难，不能悄悄降低验收标准。对长任务，建议在每个阶段形成恢复点：保存已验证产物、剩余问题和继续所需条件。

### 权限失败：不能靠“换一种说法”绕过

权限失败是系统边界，而不是提示词问题。正确处理顺序是：确认调用身份与目标资源；判断所需权限是否属于当前任务；记录被拒绝的动作和原因；寻找权限内的只读或草稿替代；如果仍无法完成，向授权人请求最小必要权限。

智能体不应尝试猜测凭据、切换到未授权账号，或通过另一个工具绕过限制。获得权限后，也应从失败步骤重新观察当前状态，因为外部数据可能已经变化。

## 案例与图解

### 文档问答智能体为什么答错了“最新版政策”

某智能体接到“汇总最新差旅政策并列出报销上限”的任务。它的长期记忆里存有去年的规则，于是直接生成答案。正确设计应当是：

1. 计划先查询政策目录，再读取标记为“当前生效”的文件。
2. 通过 resource 获取原文，记录文件版本、生效日期和来源。
3. 如需查询个人可报额度，再调用只读 tool，并显式传入授权范围内的人员标识。
4. 若个人数据工具返回未授权，继续完成公共政策摘要，把个人额度列为受阻项。
5. 评测时对照原文逐条核验，不让长期记忆覆盖当前证据。

这个案例说明：资源提供依据，工具取得动态结果，短期状态保存进度，长期记忆只提供背景，计划则约束它们的使用顺序。

## 动手实践

以“查询图书馆在借图书并生成到期提醒”为例，完成一份系统设计：

1. 分别为“查询在借图书”和“创建提醒”写工具契约，标明读取或写入权限、错误码和重试规则。
2. 画出 host、两个 client/server 连接以及 tools、resources、prompts 的位置，并写出数据不能越过的边界。
3. 把信息分到当前上下文、短期状态、长期记忆三栏，为长期记忆增加来源和失效条件。
4. 设计一个包含至少五步的计划，并加入“查询权限不足”和“创建提醒结果未知”两条失败分支。

验收标准是：任何一次失败都能被明确分类；没有权限时不会继续写入；任务中断后能根据短期状态恢复；最终提醒中的日期可以追溯到工具结果。
